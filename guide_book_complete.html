2. Technical Architecture Breakdown: The Isolated Learning Sandbox

Component 1: Immutable NixOS Scaffolding & Shell Automation

To eliminate global dependency drift and protect your local system from configuration errors, the repository rakshithskumar88/100_days_of_code uses a fully declarative NixOS environment. The configuration relies on a dedicated flake.nix file that lists exact inputs and locks system dependencies cryptographically via a flake.lock file. When you step into the workspace, direnv reads your .envrc file, automatically spins up the isolated environment, and creates a local virtual environment (.venv). This ensures your environment remains completely reproducible and system-agnostic, providing all your local binaries and servers with a single unified execution context. 

# Architectural Blueprint of flake.nix
{
  description = "Isolated Hardware-Accelerated 100 Days of Code Learning Sandbox";
  inputs = {
    nixpkgs.url = "github:nixos/nixpkgs?ref=nixos-unstable";
  };
  outputs = { self, nixpkgs }:
    let
      pkgs = nixpkgs.legacyPackages.x86_64-linux;
    in {
      devShells.x86_64-linux.default = pkgs.mkShell {
        buildInputs = with pkgs; [
          python311
          python311Packages.pip
          python311Packages.virtualenv
          ruff
          timewarrior
          nodejs
        ];
        shellHook = ''
          if [ ! -d ".venv" ]; then
            python -m venv .venv
          fi
          source .venv/bin/activate
        '';
      };
    };
}

Component 2: Two-Tier IDE Progression and Formatting Safety Constraints

The workflow splits your learning into two distinct phases to match the technical scale of the projects. For Days 1–15 (focused on core algorithmic logic), you will use both Thonny and PyCharm Community Edition side-by-side. Thonny provides a clear visual debugger that tracks execution steps and call stacks explicitly, while PyCharm builds the muscle memory needed to navigate enterprise-grade code editors. From Day 16 onwards, the environment scales completely into PyCharm to handle larger, multi-file codebases and integrated background services. To maintain high code quality without interrupting your development flow, Ruff is tied directly to PyCharm's file-save action (Ctrl + S) for immediate PEP 8 formatting. Crucially, automatic lint-fixing (ruff check --fix) is kept disabled in the editor settings. This ensures that unused import statements are never deleted out from under you during multi-day tasks or iterative coding blocks, giving you total control over when to clean up your files.

Component 3: Dual-Proxy Local AI Framework & Socratic Mentor

To provide instantaneous inline code assistance and deep pedagogical tutoring without high network latency, the workspace shifts away from standard commercial web extensions to run a specialized local inference pipeline. The pipeline splits tasks across two separate proxy servers running locally:

The FIM Autocomplete Proxy (Port 8085): The file fim_adapter.py runs a lightweight local mock server on port 8085 to intercept PyCharm's inline completion requests. This server responds to the initial synchronous IDE handshake instantly by mocking the /v1/models endpoint, which keeps background network polling away from PyCharm's primary interface thread and permanently avoids UI freezes. It maps inline context directly into special Fill-In-The-Middle formatting sequences (<|fim_prefix|>, <|fim_hole|>, <|fim_suffix|>) and streams them to a local instance of llama.cpp. By utilizing your dedicated hardware resources, this engine serves the base language model Qwen2.5-Coder-3B at blistering speeds right inside your editor as ghost text.

The Socratic Mentor Proxy (Port 9000): Interactive chat-based mentoring is handled via the DevoxxGenie extension, which is configured to route all traffic directly to socratic_proxy.py on port 9000. This server strips away any direct source code generation from the model's output, forcing the AI to act as a strict Socratic tutor that guides you through logical steps via an analytical three-tier hint progression (High-Level Concept → Abstract Pseudocode → Isolated Syntax Primitives). To guarantee this mentorship link never fails during large project evaluations, the proxy script acts as an automated token recovery chain across multiple free-tier providers, instantly catching timeout or rate-limit errors and falling back smoothly from Groq to Gemini, SambaNova, and Cerebras without dropping your chat session.

Component 5: Temporal Instrumentation & Automated Git Ledger

To track your progress effortlessly without relying on intrusive third-party database systems, the repository automates your tracking metrics locally using timewarrior and native Git hooks. The process begins with a custom start-day bash function built directly into your shell's .envrc path. When you start a block, the function initializes a background tracking event tagged with the specific day's metadata.

The automation loop completes itself when you finish your session and run a standard git commit. This action fires a custom Python script inside .git/hooks/post-commit that halts the active background timer, extracts the exact session duration, pulls your recent commit message to serve as the daily educational takeaway, and automatically appends a clean row to your central PROGRESS.md ledger file. This means your project history is always up-to-date and completely machine-readable. Whenever you query your Socratic chat, the AI can read your PROGRESS.md file via local filesystem servers to tailor its learning assistance based on your actual time spent and project milestones. 

#!/usr/bin/env python3
# Executable Artifact: .git/hooks/post-commit
import subprocess, re, os
from datetime import datetime

try:
    stop_raw = subprocess.check_output(["timew", "stop"], text=True)
except Exception:
    exit(0)

day_search = re.search(r'(Day_\d{3})', stop_raw)
time_search = re.search(r'(\d+:\d+:\d+)', stop_raw)

if day_search and time_search:
    day_idx = day_search.group(1).replace("Day_", "")
    elapsed = time_search.group(1)
    commit_log = subprocess.check_output(["git", "log", "-1", "--pretty=%s"], text=True).strip()
    date_stamp = datetime.now().strftime("%Y-%m-%d")
    
    row = f"| {day_idx} | {date_stamp} | {elapsed} | [x] Done | {commit_log} |\n"
    ledger_path = os.path.abspath(os.path.join(os.path.dirname(__file__), "../../PROGRESS.md"))
    
    with open(ledger_path, "a+") as f:
        f.write(row)
    print(f"\n[System Log]: PROGRESS.md updated for Day {day_idx} ({elapsed}).")
